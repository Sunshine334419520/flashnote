# 云同步 — 产品设计

## 概述

FlashNote 不托管服务端。用户绑定自己的 Notion 账号后，笔记自动同步到 Notion 的一个专属数据库中。换机时重新绑定同一账号即可恢复全部数据。

## 设计目标

1. **零感知**：配置一次后，同步全程自动，用户无需关心
2. **不污染 Notion 空间**：只创建一个数据库，所有笔记存为记录行
3. **防误编辑**：内部元数据封装在 `_meta` 列中（JSON），用户看到的只是普通表格
4. **可恢复**：即使 Notion 中的数据被部分修改，仍能通过元数据恢复关联

## 用户视角

### 第一步：连接 Notion

设置页 → 「云同步」→ 点击「连接 Notion」。

> 底部栏 icon 在未连接时为灰色，点击面板会引导用户前往设置页——但连接操作本身只在设置页完成。这样所有「配置类」操作统一在设置页，用户心智一致。

点击后：
1. 弹出系统浏览器，打开 Notion 授权页
2. 用户登录 Notion，选择要授权的 Workspace
3. 用户点击「允许」
4. 浏览器显示「授权成功，请返回 FlashNote」
5. FlashNote 自动完成后续配置

### 第二步：自动初始化

授权成功后，FlashNote 自动执行：
1. 在用户选择的 Workspace 中创建一个顶级 Page：「📝 FlashNote」
2. 在该 Page 下创建一个 Database（数据库）
3. 将本地全部笔记**全量上传**到该数据库
4. 底部栏云同步 icon 变为绿色 `CloudCheck`
5. 完成。用户无需任何额外操作。

### 第三步：日常使用（无感）

之后每次对笔记的增/删/改，FlashNote 自动同步到 Notion。底部栏 icon 短暂变为旋转 `RefreshCw`（同步中），完成后恢复绿色。

用户也可以在 Notion 中直接查看、搜索自己的笔记——数据库长得就像一个普通的表格。

### 换机恢复

在新电脑安装 FlashNote → 设置 → 连接同一个 Notion 账号 → FlashNote 检测到云端有数据 → 询问「检测到云端有 N 条笔记，是否恢复到本地？」→ 确认 → 全量拉取 → 完成。

## Notion 中的数据形态

### 只创建一个数据库

名为「📝 FlashNote Notes」或类似，位于一个顶级 Page 下。

### 数据库列设计

| 列名 | 类型 | 用户可见 | 说明 |
|------|------|---------|------|
| **标题** | Title（默认标题列） | ✅ | 笔记标题 |
| **内容** | Text（长文本） | ✅ | 笔记正文 |
| **类型** | Select | ✅ | API Key / Command / Credential / Bookmark / Text |
| **分类** | Select | ✅ | 用户分类 |
| **标签** | Multi-select | ✅ | 用户标签 |
| **敏感** | Checkbox | ✅ | 是否敏感内容 |
| **状态** | Select | ✅ | 已发布 / 草稿 |
| **更新时间** | Last edited time | ✅ | Notion 自动维护 |
| **`_meta`** | Text（长文本） | ⚠️ 内部使用 | JSON 元数据（见下） |

### `_meta` 列格式

```json
{
  "v": 1,
  "id": "uuid-of-note",
  "ca": "2025-07-14T10:00:00.000Z",
  "ua": "2025-07-14T12:30:00.000Z",
  "ic": true,
  "me": false,
  "sh": "cmdline",
  "td": {}
}
```

| 字段 | 含义 | 用途 |
|------|------|------|
| `v` | 同步协议版本 | 未来升级兼容 |
| `id` | FlashNote 内部 ID | 跨设备识别同一条笔记 |
| `ca` | 创建时间 | 时间戳恢复 |
| `ua` | 最后修改时间 | 增量同步 + 冲突解决 |
| `ic` | 是否已 AI 分类 | 恢复时跳过分类 |
| `me` | 是否手动编辑过 | 编辑保护 |
| `sh` | 来源标识 | `cmdline` / `quick-capture` / `import` |
| `td` | 类型特有数据 | e.g. Bookmark 的 URL、Command 的执行路径 |

> **设计意图**：`_meta` 中的字段使用短 key（减少存储），内容是 JSON blob，普通用户不会去编辑它。即便用户在 Notion 中修改了标题、内容等可见列，我们仍能通过 `_meta` 中的 `id` 和 `ua` 正确关联和处理冲突。

### 为什么不创建多个数据库？

- 一个数据库足够，用「类型」列区分笔记类型
- 减少 Notion 空间污染
- 用户查看时也更直观（一个地方看所有笔记）
- 过滤视图可以实现「只看 API Key」「只看 Command」等

## 同步行为

### 手动同步

- 底部栏面板中点击「立即同步」
- Settings 中点击「立即同步」
- 全量比较 → 增量推送/拉取

### 自动同步

| 触发条件 | 行为 | 防抖 |
|---------|------|------|
| 本地创建笔记 | 3 秒后上传 | 3s debounce |
| 本地更新笔记 | 3 秒后上传 | 3s debounce |
| 本地删除笔记 | 立即同步 | 无 |
| 应用启动 | 检查云端更新，拉取 | 启动后 5s |
| 定时轮询 | 每 5 分钟检查云端变更 | 5min interval |

### 冲突解决

以 `_meta.ua`（更新时间）为准：
1. 本地 `ua` > 云端 `ua`：本地覆盖云端
2. 云端 `ua` > 本地 `ua`：云端覆盖本地
3. 两者相等：不做任何操作

> 如果用户在 Notion 中手动编辑了内容和 FlashNote 中也编辑了，以时间戳为准。不提供手动合并——保持简单。

## 边界情况

### 用户在 Notion 中手动删除了一行
- 下次同步时，本地仍有该笔记 → 重新上传到 Notion
- 因为 `_meta.id` 在本地有记录，发现云端缺失 → 视为「需要上传」

### 用户在 Notion 中手动添加了一行
- 没有 `_meta` 列的行 → 忽略，不同步回本地
- 未来可以考虑反向同步（从 Notion 拉新笔记），但 MVP 不做

### 用户在 Notion 中修改了 `_meta` 列
- `_meta` 被破坏 → 无法关联到本地笔记 → 视为云端新增行，忽略
- 本地该笔记会被重新上传（新的 Notion 行，带正确的 `_meta`）

### 用户换了一个 Notion 账号
- 新账号没有旧数据 → 视为首次绑定 → 全量上传本地数据
- 旧账号的 Notion 数据不会自动删除（用户可以手动清理）

### Token 过期
- Notion access token 通常不过期（除非用户主动撤销）
- 如果 API 返回 401 → 状态变为「已断开」→ 提示用户重新连接

### 网络不稳定
- 同步失败 → 自动重试 3 次（间隔 5s / 15s / 60s）
- 3 次都失败 → 标记错误状态 → 下次触发时再试
- 不弹窗打扰用户，只在底部栏显示状态
